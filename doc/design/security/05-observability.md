# Observability, Audit, and Attribution

[ŌåÉ back to index](README.md)

This document describes everything the server writes to your disk or to
stderr, so you can tell at a glance (a) what you have available if you need
to investigate an incident, and (b) what is deliberately *not* recorded.

## Startup banner (stderr)

On every successful startup:

```
epimethian-mcp: connected to https://acme.atlassian.net as user@acme.com (profile: acme-corp, READ-ONLY)
```

Components:

- URL and email come from the resolved config ŌĆö they are the values the
  server actually authenticated with, not what was typed in `.mcp.json`.
- `profile: <name>` or `env-var mode` indicates which resolution path was
  used. **If you expect `profile:` and see `env-var mode`, your
  `CONFLUENCE_PROFILE` env var was not delivered to the process.**
- `READ-ONLY` appears when either the registry or `CONFLUENCE_READ_ONLY`
  env var marks the profile read-only.
- `NO-ATTRIBUTION` appears when attribution has been disabled for this
  profile.

See `src/server/confluence-client.ts:217-223`.

The banner is additionally emitted for the opportunistic tenant seal
(`sealed profile "..." to tenant "..." (cloudId ...)`) and for seal
verification warnings (`tenant_info unavailable for profile "..."`).

## Tenant echo on writes

Every successful mutating tool call returns a response that begins with:

```
Tenant: acme.atlassian.net (profile: acme-corp)
```

ŌĆ”so the AI agent sees ŌĆö and when conversation logs are reviewed, the user
sees ŌĆö exactly which tenant was written to. This is defence-in-depth: the
seal mechanism from [02-multi-tenant.md](02-multi-tenant.md) prevents the
*wrong* tenant being written to; the echo ensures that when you review the
transcript, you can confirm which one was.

## Attribution on Confluence

Unless explicitly disabled, Confluence itself carries three pieces of
attribution:

- **Page label** `epimethian-edited` added to every created or updated
  page. Makes AI-assisted edits discoverable via CQL:
  `label = epimethian-edited`.
- **Version message** on the Confluence page history: `"Updated by Claude
  Code (via Epimethian v5.4.3)"`. The "Claude Code" portion comes from the
  MCP client's declared name (via `getClientVersion()`); the version is the
  server's own `__PKG_VERSION__`.
- **Comment prefix** `[AI-generated by Claude Code via Epimethian]` on
  every created comment.

The MCP client's declared name flows through `setClientLabel()` before
being used anywhere. That helper strips characters outside the
conservative set `[A-Za-z0-9 _./()\-]` and truncates to 80 bytes, so a
malicious MCP client cannot inject ANSI escapes, newlines, or control
characters into the server's log output or into Confluence attribution.
Labels like `"Claude Code"`, `"Cursor (1.2.3)"`, `"VS Code / Continue"`
all survive unchanged; a crafted label containing `\u001b[31mŌĆ”\u001b[0m`
is sanitised to the printable residue before use.

Attribution can be disabled per profile (`attribution: false` in the
registry) or via `CONFLUENCE_ATTRIBUTION=false`. Either source disabling it
wins. This is an **opt-out**, not an opt-in ŌĆö the default leaves clear
forensic trail on the Confluence side.

See `src/server/index.ts:73-77` and the attribution handling in
`confluence-client.ts`.

## Mutation log (opt-in)

Set `EPIMETHIAN_MUTATION_LOG=true` in the server's env to enable.

- **Location:** `~/.epimethian/logs/YYYY-MM-DD.jsonl`
- **Directory mode:** `0700`; rejected outright if it is a symlink.
- **File mode:** `0600`, opened with `O_EXCL` on creation.
- **Retention:** logs older than 30 days are deleted on startup.

### What each record contains

| Field | Purpose |
| --- | --- |
| `timestamp` | ISO 8601 UTC |
| `operation` | Tool name, e.g. `update_page` |
| `pageId` | Confluence page ID |
| `oldVersion`, `newVersion` | Pre/post version numbers |
| `oldBodyLen`, `newBodyLen` | Lengths only, not content |
| `oldBodyHash`, `newBodyHash` | SHA-256 truncated to 16 hex chars |
| `clientLabel` | e.g. `Claude Code`, from MCP handshake |
| `replaceBody` | Whether the call used `replace_body: true` |
| `confirmShrinkage`, `confirmStructureLoss` | Opt-out flags passed by caller |
| `error` | Sanitised failure reason, Ōēż200 chars, first line only |

### What is *not* recorded

- **Page titles.** Titles often carry client names or project codenames;
  omitting them keeps logs from becoming a cross-tenant metadata leak if
  they are shared in incident post-mortems.
- **Page bodies.** Hashes let you prove a change occurred and whether
  two logged edits match; they do not leak content.
- **Credentials, tokens, Authorization headers.** Not touched in the
  mutation-log path.

See `src/server/mutation-log.ts`.

### What the log supports

- "Which pages did the agent touch, in what order, today?" ŌĆö sort by
  timestamp, filter by operation.
- "Did the agent use opt-out flags to bypass guards?" ŌĆö grep for
  `confirmShrinkage: true`.
- "Did this specific edit actually happen, or did we imagine it from the
  conversation?" ŌĆö compare `oldBodyHash` / `newBodyHash` with the live page.

## Profile audit log

`~/.config/epimethian-mcp/audit.log` ŌĆö simple text log of profile
add/remove events. Written with mode `0600`. Format is one entry per
line:

```
[2026-04-18T08:30:22.123Z] Removed profile "globex"
```

See `src/shared/profiles.ts:14, 171-177`.

Used to reconstruct when a profile was removed ŌĆö useful after a
"where did that profile go?" situation.

## Confluence-side audit

Because every write carries the `epimethian-edited` label and a version
message identifying the MCP client and server version, the complete change
history is reconstructable from Confluence itself without any server-side
logs. This is deliberate: even a user who never enables the mutation log
retains a full, tenant-visible audit trail as long as attribution is on.

## v6.0.0 changes

This release (see `CHANGELOG.md` and the consolidating investigations
`doc/design/investigations/investigate-agent-loop-and-mass-damage/` and
`doc/design/investigations/investigate-prompt-injection-hardening/`)
materially changes the observability surface:

### Mutation log is on by default (Track C1)

Prior behaviour: opt-in via `EPIMETHIAN_MUTATION_LOG=true`.
New behaviour: enabled by default; `EPIMETHIAN_MUTATION_LOG=false`
disables. The log remains metadata-only ŌĆö lengths, SHA-256 hashes, flag
values, client labels. Never page bodies, titles, or credentials.

Startup banner now prints a second stderr line when the log is active:

```
epimethian-mcp: mutation log enabled (/Users/u/.epimethian/logs). Set EPIMETHIAN_MUTATION_LOG=false to disable.
```

### New mutation-log fields

- `source` ŌĆö destructive-flag provenance
  (`user_request | file_or_cli_input | chained_tool_output |
  inferred_user_request`). See Track E2.
- `precedingSignals` ŌĆö injection signals detected on fenced read
  responses in the 60 s before this write. See Track D2.

### Stderr `[DESTRUCTIVE]` banner (Track C2)

Every write where `replace_body`, `confirm_shrinkage`,
`confirm_structure_loss`, or effective `confirm_deletions` was set
emits a single grep-friendly line:

```
epimethian-mcp: [DESTRUCTIVE] tool=update_page page=42 flags=replace_body,confirm_shrinkage client=claude-code
```

Lines are stderr-only ŌĆö MCP clients don't forward stderr back to the
agent, so this doesn't expand the injection surface.

### Stderr `[INJECTION-SIGNAL]` banner (Track D2)

Fired when a tenant-authored read contains suspect patterns (tool
names, destructive flag names, `IGNORE ABOVE`, ŌĆ”):

```
epimethian-mcp: [INJECTION-SIGNAL] pageId=42 field=body signals=named-tool,instruction-frame
```

Content is never logged ŌĆö only the signal classes and attribution.

### Stderr `[UNGATED]` banner (Track E4)

Printed whenever `EPIMETHIAN_ALLOW_UNGATED_WRITES=true` causes a gated
operation to proceed without elicitation on an unsupported client.
Forensic trail for deliberate opt-outs.

### Confluence version-message destructive suffix (Track C3)

Destructive writes append `[destructive: <flags>]` to the Confluence
`version.message` ŌĆö surfaces flag usage in Confluence's native
history view without dependency on the local log.
